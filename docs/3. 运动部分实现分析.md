**`m_path`（路径）从哪里来？**


## 三、运动控制核心算法 — `MoveTo()`

### 3.1 流程图

```
MoveTo(path)
│
├─ path 为空？ → return（无路径，不动）
│
├─ 已到达第一个路径点（< 0.02）？
│   └─ 移除首点，更新 target
│
├─ target 已到达（< 0.01）？
│   └─ return（完成，本轮不发指令）
│
├─ 计算目标方向角 tarAngle = atan2(vecDiff) - angle
│
├─ ◄── PID 角速度控制 ──►  Rotate(ctrl->QueryVel(tarAngle))
│
├─ 线速度方向错误（Dot(vecDiff, lineVel) <= 0）？
│   └─ Forward(0); return   ← 先停下来
│
├─ 角度偏差太大（|tarAngle| > ~90°）？
│   └─ Forward(0); return   ← 先停下来转向
│
├─ 已对准（|tarAngle| <= 0.01°）？
│   └─ Forward(999); return ← 全速前进
│
├─ 当前有速度，计算转弯弧线
│   ├─ 求转弯半径 r = |lineVel| / MAX_ROTATION_VELOCITY
│   ├─ 求圆心 center = pos ± 垂直方向 * r
│   ├─ 沿路径往前找：哪个点落在圆弧内？
│   │   └─ 找到了 → 减速转弯，Forward(newv) / Rotate(±999)
│   └─ 没找到 → 说明需要反向转弯，放弃当前弧线
│
└─ 最终：点到直线距离法
    ├─ 计算当前朝向直线到 target 的距离 a
    ├─ 计算沿朝向的投影距离 b = sqrt(c² - a²)
    └─ Forward(ctrl->QueryVel(b)) ← PID 线速度
```

### 3.2 关键设计点

**转弯弧线预判**（lines 288 – 320）：当机器人已有速度时，计算当前速度对应的转弯圆弧，沿路径向前搜索是否有路径点落在此圆弧内。如果有，说明当前速度可以平滑转弯到达；如果没有，说明需要先减速再反向转弯。

**点到直线距离法**（lines 327 – 335）：最终兜底策略——计算目标点到机器人朝向直线的垂直距离 `a` 和投影距离 `b`，用 `b` 作为 PID 控制器的输入距离。这保证了即使目标不在正前方也能合理控制速度。

## 四、速度控制器 — `SpeedCtrl`

### 4.1 核心公式

```cpp
float DecVel(float curv) {
    float t = 1.0f / FPS;           // 时间步长 = 0.02s
    float a = maxf / m;             // 最大加速度 = 最大力 / 质量
    float nextv = curv - a * t;     // 匀减速
    nextv /= (1.0f + t * 0.1);      // 额外阻尼（模拟阻力）
    return nextv;
}
```

### 4.2 预计算表

构造函数中预先计算 100 帧的减速曲线：

```
distAndVel[0] = {dist=0,      vel=0}
distAndVel[1] = {dist=0+0/FPS, vel=DecVel(0)}     ← 从 0 加速 1 帧
distAndVel[2] = {dist=...,     vel=DecVel(prev)}   ← 再加速 1 帧
...
distAndVel[99] = {..., 稳定速度}
```

运行时 `QueryVel(dist)` 二分查找：给一个剩余距离，返回该距离内能安全减速到的速度值。这本质是一个 **前馈 + 查表** 的 PID 替代方案。

### 4.3 实例化

| 控制器 | 质量/惯量 | 最大力/力矩 | 用途 |
|--------|----------|-------------|------|
| `g_radCtrl` | `ROBOT_INERTIA` | `MAX_TORQUE_BASE` | 正常态角速度 |
| `g_carryRadCtrl` | `ROBOT_CARRY_INERTIA` | `MAX_TORQUE_BASE` | 携带态角速度 |
| `g_lineVelCtrl` | `ROBOT_MASS` | `MAX_FORCE_BASE` | 正常态线速度 |
| `g_carryLineVelCtrl` | `ROBOT_CARRY_MASS` | `MAX_FORCE_BASE` | 携带态线速度 |

### 4.4 深度解析：为什么是"匀减速"

核心公式：

```
float a = maxf / m;
float nextv = curv - a * t;
```

这一步的物理依据是牛顿第二定律 `F = m·a`。当对运动反方向施加最大力 `maxf` 时，产生的减速度 `a = maxf / m`。经过 `Δt = 1/FPS = 0.02s` 一帧，速度变化量为 `a·Δt`。

**关键认知**：这不是"不给力"，而是**施加最大反向力（全力刹车）**。判题器内部每帧做相同的物理积分，所以这个模型与判题器的真实物理是对齐的。`DecVel` 建模的是"从下一帧开始全力刹车，速度如何逐帧衰减"。

### 4.5 深度解析：额外阻尼的物理依据

```
nextv /= (1.0f + t * 0.1);
```

代入 `t = 0.02`：

```
factor = 1 / (1 + 0.02 × 0.1) = 1 / 1.002 ≈ 0.998
```

每帧速度额外打 99.8 折。连续 50 帧（1 秒）后：

```
0.998^50 ≈ 0.905
```

即每秒损失约 9.5% 的速度。

**物理对应：线性阻力（viscous drag）**。假设存在与速度成正比的阻力 `F_drag = -k·v`，则：

```
dv/dt = -F_drag/m = -(k/m)·v
```

代码用的是**后向欧拉（隐式）离散**——用下一帧的速度来计算阻力：

```
v(t+Δt) = v(t) - (k/m)·v(t+Δt)·Δt
```

移项整理：

```
v(t+Δt)·(1 + (k/m)·Δt) = v(t)
v(t+Δt) = v(t) / (1 + (k/m)·Δt)
```

这就是 `nextv /= (1.0f + t * 0.1)` 的数学形式。相比前向欧拉 `v(t)·(1 - (k/m)·Δt)`，后向欧拉无条件稳定且阻尼略大（一阶展开：`1/(1+x) = 1 - x + x² - …`，比 `1 - x` 多一个正的二次项），天然带安全余量。

这里 `k/m = 0.1`。常量 `0.1` 是经验参数，作用有二：

1. **建模空气阻力 / 滚动摩擦**：判题器物理中可能隐含此类耗散项。
2. **安全余量**：让减速曲线略"肉"于理论极限，避免因浮点误差或一帧延迟而冲出目标点。

### 4.6 深度解析：前馈 + 查表究竟在算什么

#### 预计算表的构造（刹车逆推）

构造函数中递推 100 帧：

```
distAndVel[0] = {dist=0, vel=0}                              // 终点：速度为 0，剩余距离为 0
for i = 1 .. 99:
    dist[i] = dist[i-1] + vel[i-1] / FPS                     // 以上一帧速度匀速前行的距离
    vel[i]  = InversIncFunc(DecVel, vel[i-1])                // 反解：什么初速度刹车一帧后恰好变为 vel[i-1]？
```

`InversIncFunc(DecVel, vel[i-1])` 用二分法求解方程 `DecVel(v) = vel[i-1]`，即找到 `v` 使得全力刹车一帧后速度降为 `vel[i-1]`。

**核心思想**：从终点（v=0, d=0）反向递推，算出"在距离终点 `dist[i]` 处，速度为 `vel[i]` 时，全力刹车恰好停在终点"。这张表记录了 100 帧的刹车逆推轨迹。

#### 运行时查表（QueryVel）

```cpp
auto it = lower_bound(table, dist);   // 二分查找：第一个 braking_distance ≥ 剩余距离的条目
return prev(it)->vel;                 // 取前一档的速度（保守：刹车距离不超过剩余距离）
```

**翻译成人话**："我还剩 `dist` 米到目标，表中哪一档的刹车距离刚好不超过这个距离？取那档对应的速度。"

#### 为什么是"前馈"

与 PID 的本质区别：

| | PID | 前馈 + 查表 |
|---|---|---|
| 输入 | 误差（当前位置 - 目标位置） | 剩余距离 |
| 计算方式 | `Kp·e + Ki·∫e + Kd·de/dt` | 物理模型反解 |
| 依赖 | 需要在线调参 | 只需知道质量和最大力 |
| 本质 | 反馈校正（事后补救） | 物理预测（事前限制） |

`QueryVel(dist)` 直接回答"这个距离下最多能跑多快"，不依赖历史误差积分，也没有振荡风险。这不是 PID 的替代品——`MoveTo()` 中角速度控制依然用 PID（见 3.1 节），线速度终点对准也用了 PID（`QueryVel(b)` 传入的是投影距离 `b`）。**前馈表负责"速度上限"（安全边界），PID 负责"当前该给多少力"（实时调节）**。

### 4.7 深度解析：角速度为什么不在线速度预计算表中

分两个层面理解。

#### 角速度有自己独立的表

```cpp
SpeedCtrl g_radCtrl(ROBOT_INERTIA, MAX_TORQUE_BASE);       // 角速度刹车表
SpeedCtrl g_lineVelCtrl(ROBOT_MASS, MAX_FORCE_BASE);       // 线速度刹车表
```

两个表是同一套 `SpeedCtrl` 模板，只是物理参数不同。线速度用质量和力，角速度用惯量和力矩。角速度不是"没考虑"，而是有自己独立的一张刹车距离（角度）表。

#### 线速度和角速度物理解耦

判题器的物理模型中，前进力只改变线速度，力矩只改变角速度，两者互不耦合。因此：

- 线速度的刹车距离 = f(线速度, 质量, 最大力)，与当前角速度无关。
- 角速度的刹车角度 = f(角速度, 惯量, 最大力矩)，与当前线速度无关。

转弯时线速度方向和位置同时变化的耦合，不在 `SpeedCtrl` 层面处理——它在 `MoveTo()` 的"转弯弧线预判"环节（见 3.2 节）单独计算。`SpeedCtrl` 只回答一个标量子问题：给定剩余距离，最高安全速度是多少。

## 五、冲突处理 — `MovePath()`

```
MovePath()
│
├─ Map::CheckConflict() → confRobot 位表
│
├─ 有冲突？
│   ├─ 标记冲突双方 beConflicted = FPS (50 帧)
│   ├─ 用 Dijkstra 重新搜索无冲突路径
│   ├─ 选一个最优绕行点 select
│   │   └─ 找不到 → forcePri = 1（强制优先）
│   ├─ 生成新路径 m_path = dij.MoveToDiscrete(stat, select)
│   └─ 重新 OptimizePath + CheckConflict
│
└─ MoveTo(m_path) → 执行
```

冲突后的限速：`Forward()` 中检查 `beConflicted > 0 && forcePri == 0 && v > 4` 时，将速度限制在 4 m/s，降低碰撞风险。
