# 1. Baseline 多线程设计文档

> 基于张元龙老师示例代码（SimpleDemo）的多线程架构分析  
> 日期：2026-05-20

---

## 一、整体架构

系统采用 **双线程** 模型：

```
┌─────────────────────────────┐     ┌─────────────────────────────┐
│        主线程 (MainLoop)      │     │    后台线程 (ThreadMain)      │
│                             │     │                             │
│  Input() ──► SwapInput() ───┼──┬──┼──► SwapInput(nullptr)      │
│             (投递雷达数据)    │  │  │     (获取雷达数据)            │
│                             │  │  │                             │
│  UpdateResult() ◄───────────┼──┼──┼── m_result (加锁写入)       │
│             (拉取最新结果)    │  │  │                             │
│                             │  │  │  UpdateLidar()              │
│  Dispatch()                 │  │  │  UpdateMap()                │
│    GreedyBuy/GreedySell     │  │  │  UpdateDijkstra()           │
│    MovePath()               │  │  │                             │
│                             │  │  │                             │
│  Finish()                   │  │  │                             │
│                             │  │  │                             │
│  while(IsWorking()) sleep ──┼──┼──┼── m_working = false        │
│        (等待后台完成)         │  │  │                             │
└─────────────────────────────┘     └─────────────────────────────┘
```

**设计目标**：主线程以 50 FPS 运行，每帧仅 20ms 时限。将耗时的地图更新和 Dijkstra 计算卸载到后台线程，避免主线程跳帧。

## 二、线程间通信机制

系统使用 **两种不同的同步原语**，分别处理两个方向的数据流：

### 2.1 主→后台：无锁原子交换（`SwapInput`）

```cpp
// BackgroundThread.h

// 该函数用于两个线程之间投递最新的雷达数据
// 如果一个已投递数据没有被另一个线程获取，那么会被第二次投递替换
Input *SwapInput(Input *input)
{
    return std::atomic_exchange(&m_lidarInput, input);
}
```

**工作流程**：

```
时间线 ──────────────────────────────────────────────────────►

主线程:
  Frame 1:  new Input_A ──► SwapInput(A)  返回 nullptr
  Frame 2:  new Input_B ──► SwapInput(B)  返回 nullptr  
  Frame 3:  new Input_C ──► SwapInput(C)  返回 nullptr  (后台还在忙)
  Frame 4:  input≠null ──► 复用 Input_C ◄── 返回 Input_C (后台没来得及消费)

后台线程:
  SwapInput(nullptr) ──► 得到 Input_A  ──► 处理 ──► delete
  SwapInput(nullptr) ──► 得到 Input_B  ──► 处理 ──► delete
                         (忙...)
  SwapInput(nullptr) ──► 得到 Input_C  ──► 处理 ──► delete
```

**关键语义**：

| 行为 | 含义 |
|------|------|
| `SwapInput(ptr)` 返回 `nullptr` | 后台已消费上一份数据，队列为空 |
| `SwapInput(ptr)` 返回非 `nullptr` | 后台未及时消费，返回旧指针供复用 |
| `SwapInput(nullptr)` 返回非 `nullptr` | 有新数据到达 |

`std::atomic_exchange` 保证 **无锁、无等待** 的指针交换，主线程投递永远不会阻塞。

### 2.2 后台→主：互斥锁保护（`m_lock`）

```cpp
// 后台线程写入
void BackgroundThread::ThreadMain()
{
    Output result;
    // ... 激光处理、地图更新、Dijkstra 计算 ...
    {
        std::lock_guard<mutex> lck(m_lock);
        m_result = move(result);   // 原子地替换结果
    }
}

// 主线程读取
bool BackgroundThread::UpdateResult()
{
    m_lock.lock();
    m_returnResult = m_result;     // 拷贝 shared_ptr（引用计数+1）
    m_lock.unlock();

    if (!m_returnResult.map || !m_returnResult.workbenchDij) {
        return false;              // 后台尚未产出有效结果
    }

    // 锁外操作：重置 Dijkstra 对象的 map 指针
    for (auto &it : *m_returnResult.workbenchDij) {
        it.ResetMap(m_returnResult.map.get());
    }
    return true;
}
```

**为什么这里用锁而不是无锁**：

`Output` 包含 `shared_ptr<Map>` 和 `shared_ptr<vector<Dijkstra>>`，不是简单的指针。`shared_ptr` 的拷贝/赋值需要原子增减引用计数，但整个 Output 的替换不是原子的——必须加锁保证 `map` 和 `workbenchDij` 始终配对（都来自同一次计算）。

## 三、数据流详解

### 3.1 主线程每帧执行流程

```
MainLoop()
│
├─ Input()                          // 读取 stdin 数据
│   ├─ 填充 Input 结构体
│   │   ├─ robot[i].pos, .angle    // 机器人位姿
│   │   ├─ robot[i].lidar[360]     // 激光雷达数据
│   │   └─ workbench[]             // 工作台信息
│   │
│   ├─ SwapInput(input)            // 投递给后台线程
│   │
│   └─ Frame 1 时: Init()          // 启动后台线程
│
├─ UpdateResult()                  // 从后台拉取结果
│   └─ 拷贝 m_result ─► m_returnResult
│
├─ Dispatch()                       // 策略计算（仅在 result 有效时）
│   ├─ ResetConflict()
│   ├─ LastSellStrategy()
│   ├─ GreedySell()                // 贪心卖出
│   ├─ GreedyBuy()                 // 贪心买入
│   ├─ Attack()                    // 攻击对手
│   └─ MovePath()                  // 执行移动
│
├─ while (IsWorking() && 时间<12ms)
│       sleep(1ms)                 // 等待后台完成（最多等 12ms）
│
└─ Finish()                        // 输出本帧指令
```

### 3.2 后台线程执行流程

```
ThreadMain()
│
└─ while (!m_exited)
    │
    ├─ SwapInput(nullptr)          // 尝试获取主线程投递的数据
    │   └─ 若为 nullptr ─► sleep 1ms ─► continue
    │
    ├─ m_working = true
    │
    ├─ UpdateLidar(input, &result)     // 激光雷达 → 可通行/障碍物位表
    │
    ├─ if (地图有更新)
    │   ├─ UpdateMap(input, &result)   // 重新生成地图
    │   └─ UpdateDijkstra(...)         // 重新计算所有工作台的最短路
    │
    ├─ { lock_guard: m_result = move(result) }  // 发布结果
    │
    ├─ delete input                  // 释放主线程传来的数据
    │
    └─ m_working = false
```

## 四、内存所有权模型

```
                  SwapInput() 原子交换
  主线程 ─────────────────────────────────────► 后台线程
    │                                              │
    │  ① new Input ────► m_lidarInput ◄── nullptr  │
    │                    (原子指针)                  │
    │                                              │
    │  ②        ◄──── SwapInput 返回 ────►          │  ③ 获取 Input*
    │            (旧指针或nullptr)                    │     处理完毕后 delete
    │                                              │
    │  ④ 若返回非空: 复用旧指针                        │
    │     若返回空:   下次 new                        │
```

**所有权规则**：

1. `SwapInput` 调用后，**调用方交出传入指针的所有权**，同时获得返回指针的所有权
2. 同一 `Input` 对象 **永远不会被两个线程同时访问**
3. 主线程的 `input` 指针若非空，说明后台未消费上一帧数据，主线程直接覆写该内存（安全，因为后台不持有此指针）
4. 后台线程总是 `delete` 其获取的 `Input*`，释放所有权

## 五、潜在问题分析

### 5.1 偶然卡死/跳帧的可能原因

#### 原因一：后台线程计算超时

```cpp
// Main.cpp:1139
while (BackgroundThread::Instance().IsWorking() 
       && beginTime + 12 < GetMillTime())
    std::this_thread::sleep_for(std::chrono::milliseconds(1));
```

主线程最多等后台 **12ms**。若后台 Dijkstra 计算耗时超过 12ms（复杂地图 + 多工作台），主线程不等了直接 `Finish()`。此时：
- `m_returnResult` 仍是**上一帧的旧数据**
- 地图可能有更新但未同步到主线程
- 主线程基于过期数据决策，可能导致机器人碰撞或路径错误

#### 原因二：`UpdateResult()` 中 `m_lock` 竞争

```cpp
bool BackgroundThread::UpdateResult()
{
    m_lock.lock();           // ① 获取锁
    m_returnResult = m_result; // ② 拷贝
    m_lock.unlock();         // ③ 释放锁
    // ... ④ 锁外操作 ...
}
```

后台线程在 ② 期间可能正持有 `m_lock` 写入 `m_result`。此时主线程阻塞在 `m_lock.lock()`（①），直到后台释放。在极端情况下（后台在锁内做大量操作），主线程可能阻塞数毫秒。

但当前实现中，后台在锁内仅做 `m_result = move(result)`（一条赋值语句），阻塞时间极短，**不是瓶颈**。

#### 原因三：`m_working` 标志的 TOCTOU 竞态

```
主线程检查 IsWorking() → false → 跳过等待 → 调用 Finish()
                                    ↑
后台线程正好在这里: m_working = true → 开始处理新数据
```

此时主线程已结束本帧，后台的本次计算结果将在 **下一帧** 才被 `UpdateResult()` 读取。这会导致本帧使用了旧数据。

**影响**：单帧延迟，不会累积。下一帧能拿到更新结果。

#### 原因四：`Dijkstra::ResetMap()` 修改共享对象

```cpp
// UpdateResult() 中，锁已释放
for (auto &it : *m_returnResult.workbenchDij) {
    it.ResetMap(m_returnResult.map.get());  // 修改 Dijkstra 对象
}
```

`m_returnResult.workbenchDij` 和 `m_result.workbenchDij` 是**同一个 `shared_ptr`** 指向的同一个 vector。`ResetMap` 修改 vector 中的 `Dijkstra` 对象时，后台线程可能正在创建新的 Output（不影响旧对象），也可能完全空闲。

**但存在一个微妙窗口**：后台线程在下一轮 `m_result = move(result)` 时，旧 `shared_ptr` 的引用计数从 2 降为 1（主线程还持有 `m_returnResult`）。只有当主线程下一帧 `m_returnResult = m_result` 覆盖后，旧对象才被析构。在此之前，两个 `shared_ptr` 共享同一份 Dijkstra 数据，而主线程在修改它。

**结论**：后台不会并发访问旧 Dijkstra（因为后台只访问 `m_result`，不访问 `m_returnResult`），所以安全。

---

### 5.2 设计上的不完美之处

| 问题 | 严重程度 | 说明 |
|------|----------|------|
| 无超时兜底 | ⚠️ 中 | 后台计算超时后，主线程用旧数据继续，无感知 |
| UpdateResult 不检查 m_working | ⚠️ 中 | 可能读到不完整的结果（`map` 已更新但 `workbenchDij` 未更新） |
| while 等待无退出条件 | 🔵 低 | 如果 `m_working` 卡在 true（线程异常），12ms 后退出，安全 |
| 静态地图分支无意义 | 🔵 低 | `SetMapData` 最后设置 `m_staticMap = true`，但已在函数开头设置过 |

## 六、总结

该多线程设计采用 **生产者-消费者** 模式的变体：

- **数据投递**：无锁 `atomic_exchange` 指针交换，保证主线程零阻塞
- **结果回传**：`mutex` + `shared_ptr`，保证 map 与 Dijkstra 的原子性配对
- **容错**：12ms 超时等待 + `UpdateResult` 空值检查，防止后台异常导致主线程崩溃

核心优点是指针交换的无等待特性，确保主线程在 50 FPS 下不会因为数据投递而阻塞。存在的风险主要在后台计算超时场景下使用过期数据，以及 `m_working` 标志的判断窗口可能导致单帧延迟。
